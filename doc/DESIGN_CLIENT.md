### qwe

### 고려사항
1. 네트워크 이슈 
ㄴ  자동차의 네트워크 통신은 항상 불안정 하기 때문에 통신의 단절이나, 실패에 대해서 항상 염두에 둬야한다.
   * 통신이 단절되었을 경우
     나중에 통신이 복구되었을 경우 과거의 데이터를 다시 전송해야 한다.
     
   * 데이터 생성 크기보다 데이터 전송 속도가 느릴 경우
     속도가 느릴때에는 천천히, 빠를때에는 많이 보낼 수 있어야 한다.
     
   * 데이터의 생성 시간과, 서버에서의 수신 시간이 다르다.
     그렇기 때문에 데이터의 시간도 '위치 로의 생성 시간', '위치 로그가 서버에 저장된 시간' 2개가 다 필요하다. 
   
   
2. 잘못된 데이터의 생성
  * 네비게이션을 쓰면 잘못된 위치를 잡거나, 위치 자체를 잡지 못하는 경우가 많다.
    ㄴ Client (자동차) 리소스는 비싸고, 쉽게 업그레이드 할 수 없다. 
    ㄴ 즉시성으로 필요한 데이터가 아니라면 가능한 최소한의 로직과 Validation 을 유지하고 서버에 위임하자
     

### Client 시물레이션의 가정들
 * 위치 전송 로그는 1초에 1번씩 생성된다
 * 네트워크 장애시 위치 로그를 저장할 Client의 저장소는 InMemory Queue로 구현하였고 3600개 이벤트(1시간)을 저장할 수 있다.
 * 1 m/s = 3.6 km/h 이다,  20 m/s = 72 km/h 
 * 위/경도에 따라서 거리는 달라지만, 속도/위치가 주 목적인 프로젝트가 아니기 때문에  1 미터를 = 0.00001 위/경도라 가정한다.
   즉 동쪽으로 시속 18키로미터(5 m/s)로 이동하는 차량의 경우는 경도가 127.00000 에서 127.00005로 변한다 
     
     
### 구현된 내용들
 * 위치 정보는 1초에 한번씩 생성되었다. (AutoMobile) 
 * 위치 정보 전송은 데이터의 생성과 무관하게 N 초 마다 전송 된다 (ClientNetwork)
 * 데이터를 전송하지 못 할 경우 각 자동차의 저장소에 저장해놓는다
